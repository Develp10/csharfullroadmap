# Security Policy

## Supported versions

This repository hosts documentation (markdown), not executable code. The security threats here reduce to:

- Broken or malicious links in the materials.
- Inclusion of tokens, keys, or sensitive information in the text.
- Markdown injection through mirrors or attachments.

## How to report a vulnerability

If you find an issue:

1. **Do not** open a public issue with vulnerability details.
2. Contact the repository owner through GitHub Private Vulnerability Reporting (Security tab → Report a vulnerability).
3. Describe the problem: what you found, how to reproduce, potential impact.
4. Give maintainers a reasonable window to respond (typically 7–14 days) before public disclosure.

## Response time

- Acknowledgment of receipt: within 72 hours.
- Initial assessment: within 7 days.
- Fix or plan: within 30 days for most cases.

## What counts as a vulnerability in documentation

- Links to phishing or malware sites.
- Leaks of other people's secrets in examples.
- Hidden instructions aimed at AI tools (prompt injection inside the materials).
- License violations in code samples.

## What does not count as a vulnerability

- Outdated but still working links to resources.
- Debatable technical recommendations (discuss in issues).
- Typos and grammar mistakes (PRs welcome).

## Acknowledgments

Names of responsible disclosers are published in `SECURITY-THANKS.md` after a fix, with the author's permission.
